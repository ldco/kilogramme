## Обзор

Инфраструктура разработки построена вокруг **Kilo Code** — терминального AI-ассистента с открытым исходным кодом, который связывает воедино языковую модель (LLM), протокол интеграции инструментов (MCP) и систему семантической индексации кода (Code Indexing). Такая архитектура позволяет AI-ассистенту не просто генерировать текст, а **полноценно работать с кодовой базой**: читать файлы, выполнять shell-команды, искать код по смыслу, запрашивать актуальную документацию, взаимодействовать с веб-страницами и базой данных.

## Kilo Code — ядро системы

**Версия:** 7.3.45  
**Тип:** CLI (Command Line Interface) приложение, бинарный файл (~144 МБ)  
**Лицензия:** Open Source  
**Сайт:** [kilo.ai](https://kilo.ai)

### Что это

Kilo Code — это терминальный интерфейс для взаимодействия с языковыми моделями (LLM), специализированный для задач разработки. В отличие от веб-чатов (ChatGPT, Claude.ai), Kilo Code имеет прямой доступ к файловой системе проекта, shell-окружению, git-репозиторию и подключённым инструментам.

### Зачем он нужен

| Возможность | Зачем |
|---|---|
| **Прямой доступ к коду** | Чтение, редактирование, поиск по файлам проекта без копирования кода в буфер обмена |
| **Shell-интеграция** | Выполнение команд (npm, git, docker, тесты) и анализ вывода |
| **Мульти-агентность** | Специализированные агенты: архитектор, код-ревьюер, тестировщик, документатор |
| **MCP-интеграция** | Подключение внешних инструментов через единый протокол |
| **Индексация кода** | Семантический поиск по кодовой базе (понимание смысла, а не текста) |
| **LSP** | Языковой сервер для навигации по коду (переход к определению, поиск использований) |

### Как работает

Kilo Code запускается как TUI (Text User Interface) в терминале. Пользователь вводит задачу на естественном языке — ассистент выполняет её, используя доступные инструменты: читает файлы, правит код, запускает команды, ищет документацию. Каждое действие проходит через систему разрешений (allow/ask/deny), настраиваемую в `kilo.json`.

### Конфигурация

Глобальная конфигурация хранится в `~/.config/kilo/kilo.json`. Проектная — в `.kilo/kilo.json`. Приоритет слияния: проектный конфиг переопределяет глобальный (deep merge).

## DeepSeek API — LLM-провайдер

**Модель:** `deepseek-v4-pro` (DeepSeek V4 Pro)  
**Провайдер:** DeepSeek (deepseek.com)

### Что это

DeepSeek API предоставляет доступ к языковой модели DeepSeek V4 Pro — одной из самых производительных LLM, оптимизированной для генерации кода и решения сложных технических задач.

### Почему DeepSeek

- **Высокое качество кодогенерации** — DeepSeek V4 Pro входит в число лидеров по бенчмаркам кода (HumanEval, MBPP, SWE-bench)
- **Большой контекст** — поддержка длинных диалогов без потери связности
- **Поддержка thinking (reasoning)** — модель способна к пошаговому рассуждению перед выдачей ответа, что критично для сложных архитектурных задач
- **Экономическая эффективность** — конкурентная стоимость относительно аналогов (Claude, GPT-4)
- **Открытая экосистема** — DeepSeek публикует исследования и веса моделей

### Роль в системе

DeepSeek API — это «мозг» всей системы. Kilo Code отправляет запросы (prompt + контекст) в DeepSeek API, получает ответы и транслирует их в действия (редактирование кода, выполнение команд, поиск). Без LLM-провайдера Kilo Code — просто терминальная оболочка без интеллекта.

## MCP — Model Context Protocol

**MCP (Model Context Protocol)** — это открытый протокол, стандартизирующий способ подключения внешних инструментов и источников данных к AI-ассистентам. Его можно представить как «USB-порт для AI»: единый интерфейс, через который модель получает доступ к базам данных, API, файловым системам, веб-браузерам и другим сервисам.

### Архитектура MCP

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  Kilo Code   │────▶│  MCP Server  │────▶│  SQLite DB   │
│  (клиент)    │     │  (sqlite)    │     │  (файл .db)  │
└──────────────┘     └──────────────┘     └──────────────┘
       │
       │              ┌──────────────┐     ┌──────────────┐
       └─────────────▶│  MCP Server  │────▶│  Контекст7   │
                      │  (context7)  │     │  API         │
                      └──────────────┘     └──────────────┘
       │
       │              ┌──────────────┐     ┌──────────────┐
       └─────────────▶│  MCP Server  │────▶│  Chromium    │
                      │  (puppeteer) │     │  (браузер)   │
                      └──────────────┘     └──────────────┘
```

Каждый MCP-сервер — это отдельный процесс, запускаемый Kilo Code. Сервер предоставляет набор «инструментов» (tools), которые модель может вызывать через стандартизированный JSON-RPC интерфейс.

### Context7 — документация по требованию

**Пакет:** `@upstash/context7-mcp`  
**Назначение:** Предоставление актуальной документации и примеров кода по любым библиотекам и фреймворкам.

#### Зачем это нужно

LLM-модели обучаются на данных с фиксированной датой отсечки (knowledge cutoff). Когда разработчик использует новую версию библиотеки или редкий фреймворк, модель может не знать актуального API. Context7 решает эту проблему: модель запрашивает документацию «на лету» и получает релевантные примеры кода из официальных источников.

#### Как работает

1. **Resolve Library ID** — определение идентификатора библиотеки в базе Context7 (например, `/vercel/next.js`)
2. **Query Docs** — запрос документации по конкретному вопросу (например, «как настроить middleware в Next.js 15»)
3. **Возврат примеров** — Context7 возвращает сниппеты кода и описания из официальной документации

#### Что покрывает

Более 20 000 библиотек: React, Next.js, Drizzle ORM, Zod, Nuxt, Tailwind, Prisma, и многие другие.

### SQLite MCP — прямой доступ к базе данных

**Пакет:** `mcp-sqlite-server`  
**Назначение:** Прямое взаимодействие с SQLite-базами данных — чтение схемы, выполнение запросов, анализ данных.

#### Зачем это нужно

В проекте используется SQLite (через Drizzle ORM). Во время разработки ассистенту часто нужно:
- Проверить структуру таблиц (схему)
- Проверить данные в базе (миграции, seed)
- Отладить запросы
- Проверить состояние индексации (Kilo хранит метаданные в `kilo.db`)

Без MCP ассистенту пришлось бы каждый раз запускать `sqlite3` через shell и парсить текстовый вывод. MCP даёт структурированный доступ: `sqlite_query`, `sqlite_schema`, `sqlite_explain`.

#### Инструменты

| Инструмент | Назначение |
|---|---|
| `sqlite_query` | Выполнение SELECT/Pragma запросов |
| `sqlite_schema` | Получение полной схемы БД (таблицы + колонки) |
| `sqlite_table_info` | Детальная информация о конкретной таблице |
| `sqlite_explain` | EXPLAIN QUERY PLAN для оптимизации |
| `sqlite_list_databases` | Поиск файлов .db в директории |

#### Безопасность

По умолчанию — только чтение (`readonly: true`). Запись требует явного разрешения.

### Puppeteer MCP — взаимодействие с веб-страницами

**Пакет:** `puppeteer-mcp-claude`  
**Назначение:** Автоматизация браузера — навигация по страницам, скриншоты, клики, извлечение текста.

#### Зачем это нужно

В процессе разработки ассистенту может потребоваться:
- **Проверить вёрстку** — сделать скриншот страницы и визуально оценить результат
- **Протестировать UI** — заполнить форму, нажать кнопку, проверить результат
- **Извлечь данные** — получить текст со страницы документации, которая не отдаёт API
- **Отладить фронтенд** — проверить наличие элементов на странице, их состояние

#### Инструменты

| Инструмент | Назначение |
|---|---|
| `puppeteer_navigate` | Переход по URL |
| `puppeteer_screenshot` | Скриншот страницы (в том числе full-page) |
| `puppeteer_click` | Клик по CSS-селектору |
| `puppeteer_type` | Ввод текста в поле |
| `puppeteer_get_text` | Извлечение текста элемента |
| `puppeteer_evaluate` | Выполнение JavaScript на странице |
| `puppeteer_wait_for_selector` | Ожидание появления элемента |

## Индексация кода — Code Indexing

### Зачем нужна индексация кода

Обычный текстовый поиск (grep, ripgrep) ищет **точные совпадения строк**. Это бесполезно, когда нужно найти код, решающий определённую задачу, но написанный с другими именами переменных или в другом стиле.

**Пример:** Нужно найти все места, где происходит валидация email. Grep найдёт только строки с `email` и `validate`. Но не найдёт код, где проверка написана через регулярное выражение без слова `email`, или через библиотеку `z.string().email()`.

**Code Indexing** превращает фрагменты кода в **векторные представления** (эмбеддинги) — числовые массивы, в которых семантически похожий код находится близко друг к другу. Это позволяет искать **по смыслу**, а не по тексту.

#### Фаза 1: Сканирование

Directory Scanner обходит все файлы проекта, исключая:
- `node_modules/`, `.git/`, `dist/`, `.output/`
- Бинарные файлы и медиа
- Файлы, не соответствующие поддерживаемым расширениям (`.ts`, `.vue`, `.css`, `.js`, `.py`, и др.)

Для каждого файла вычисляется хеш — это позволяет пропускать неизменённые файлы при повторном сканировании (инкрементальное обновление).

#### Фаза 2: Парсинг и извлечение блоков

Code Parser (на базе Tree-sitter) разбирает исходный код на структурные блоки:
- **Функции** — `function`, `const x = () => {}`
- **Классы** — `class Foo { ... }`
- **Методы** — функции внутри классов
- **Vue компоненты** — `<script setup>`, `<template>`
- **CSS правила** — селекторы с объявлениями

Каждый блок сохраняется с метаданными: путь к файлу, номер строки, тип блока.

#### Фаза 3: Эмбеддинг (векторизация)

Каждый блок кода отправляется в Voyage AI API, который возвращает **вектор** — массив из 1024 чисел (для `voyage-code-3`). Близкие в векторном пространстве блоки кода семантически похожи.

#### Фаза 4: Хранение

LanceDB — это векторная база данных, встроенная в процесс (embedded). Она хранит векторы на диске в колоночном формате Lance, оптимизированном для:
- **Быстрого поиска ближайших соседей** (ANN — Approximate Nearest Neighbors)
- **Фильтрации по метаданным** (тип файла, путь)
- **Инкрементальных обновлений** (добавление/удаление отдельных векторов без перестроения индекса)

#### Фаза 5: Поиск (рантайм)

Когда ассистент ищет код:
1. Поисковый запрос (например, «где обрабатывается ошибка аутентификации») векторизуется через Voyage AI
2. LanceDB находит N ближайших векторов (косинусное расстояние)
3. Результаты фильтруются по порогу релевантности (`scoreThreshold`)
4. Ассистент получает список файлов и строк с релевантным кодом

#### File Watcher

После первичного сканирования запускается File Watcher, который отслеживает изменения в файловой системе и автоматически переиндексирует изменённые файлы.

### Voyage AI — провайдер эмбеддингов

**Модель:** `voyage-code-3`  
**Размерность вектора:** 1024  
**API:** REST API (`https://api.voyageai.com/v1/embeddings`)  
**Сайт:** [voyageai.com](https://www.voyageai.com)

#### Что такое Voyage AI

Voyage AI — это специализированный провайдер embedding-моделей (в отличие от OpenAI, Anthropic, которые делают универсальные LLM). Их модели оптимизированы исключительно для задачи преобразования текста в векторы.

#### Почему Voyage AI, а не OpenAI embeddings

| Критерий | Voyage AI (`voyage-code-3`) | OpenAI (`text-embedding-3-small`) |
|---|---|---|
| **Специализация** | Специально для кода | Универсальные (текст, код, поиск) |
| **Качество на коде** | Лидер по бенчмаркам (COSCI, CoIR) | Хорошее, но не лучшее для кода |
| **Размерность** | 1024 | 1536 |
| **Стоимость** | Конкурентная | Конкурентная |
| **Задержка** | Низкая | Низкая |
| **BYOK** | Да (собственный ключ) | Да |

`voyage-code-3` специально обучена на парах (код, документация) и (код, запрос), что делает её особенно точной для поиска по кодовой базе.

#### Аутентификация

API-ключ Voyage (`pa-...`) хранится в конфигурации Kilo Code. Каждый запрос на эмбеддинг подписывается заголовком `Authorization: Bearer <key>`.

### LanceDB — векторное хранилище

**Формат:** Lance (columnar, оптимизирован для ML-нагрузок)  
**Тип:** Embedded (встроенная в процесс, без отдельного сервера)  
**Расположение:** `~/.cache/kilo/packages/@lancedb/`

#### Что такое LanceDB

LanceDB — это векторная база данных нового поколения, построенная на колоночном формате Lance (альтернатива Parquet, но с оптимизациями для случайного доступа и векторного поиска).

#### Почему LanceDB, а не Qdrant/Pinecone/Chroma

| Критерий | LanceDB | Qdrant | Pinecone |
|---|---|---|---|
| **Развёртывание** | Встроенная (ноль инфраструктуры) | Отдельный сервер (Docker) | Облачный сервис |
| **Производительность** | Высокая (колоночный формат) | Высокая | Высокая |
| **Зависимости** | Нет (файлы на диске) | Docker / бинарный сервер | Интернет + аккаунт |
| **Стоимость** | Бесплатно | Бесплатно (self-hosted) | Платно |
| **Подходит для** | Локальная разработка, один проект | Продакшен, многопользовательский доступ | Продакшен, управляемый сервис |

Для локальной разработки LanceDB — идеальный выбор: не требует отдельного сервера, хранит данные в файлах проекта, мгновенно запускается.

### Полный цикл работы индексации

```
СТАРТ KILO CODE
       │
       ▼
┌─────────────────┐
│  Чтение конфига │  kilo.json → indexing.enabled: true
│  kilo.json      │               provider: voyage     
└────────┬────────┘               model: voyage-code-3
         │                        vectorStore: lancedb
         ▼
┌─────────────────┐
│  Валидация       │  Проверка API-ключа Voyage (тестовый запрос)
│  конфигурации    │  Проверка доступности LanceDB
```

## RAG — Retrieval-Augmented Generation

### Что такое RAG

**RAG (Retrieval-Augmented Generation)** — архитектурный паттерн, при котором перед отправкой запроса в LLM система извлекает релевантные фрагменты из внешней базы знаний и подставляет их в промпт как контекст. LLM не «знает» этих данных из обучения (knowledge cutoff) — она видит их только в момент запроса.

```
Пользователь → Запрос → Поиск по базе знаний → Релевантные фрагменты → Промпт + фрагменты → LLM → Ответ
              │                                ↑
              └────── Векторизация запроса ────┘
```

**Ключевое свойство RAG:** модель получает актуальный, специфичный для проекта контекст, которого не было в обучающей выборке. Это решает проблему устаревших знаний и галлюцинаций на неизвестных данных.

### RAG в текущем стеке: гибридная реализация

Классический RAG и текущий Code Indexing — это **одна и та же архитектура**, применённая к разным источникам данных. В стеке Kilo Code RAG реализован в двух независимых retrieval-слоях:

| Шаг RAG | Классическая реализация | В этом стеке (домен: код) | В этом стеке (домен: документация) |
|---|---|---|---|
| **Ingest** (загрузка) | Парсинг PDF/HTML/Markdown | Tree-sitter-парсинг функций, классов, Vue-компонентов | Crawl официальной документации библиотек |
| **Эмбеддинг** | `text-embedding-3-large` | `voyage-code-3` (1024d, специализация на коде) | Серверная векторизация Context7 |
| **Хранение** | Pinecone / Qdrant / Weaviate | LanceDB (embedded, колоночный Lance-формат) | База знаний Context7 (Upstash) |
| **Retrieval** | ANN-поиск по косинусному расстоянию | `semantic_search` → LanceDB ANN | `context7_query-docs` → семантический поиск по документации |
| **Augmentation** | Подстановка чанков в промпт | Найденные файлы + строки кода → контекст LLM | Сниппеты кода + описания API → контекст LLM |
| **Generation** | GPT-4 / Claude / Llama | DeepSeek V4 Pro | DeepSeek V4 Pro |

#### Retrieval-слой 1: Код проекта (Code Indexing)

Векторизует и индексирует всю кодовую базу проекта. Позволяет задавать вопросы вроде «где обрабатывается ошибка аутентификации» на естественном языке и получать точные ссылки на файлы и строки. Подробно описан в [разделе индексации](#индексация-кода--code-indexing).

#### Retrieval-слой 2: Документация библиотек (Context7)

Извлекает актуальную документацию и примеры кода «на лету» по 20 000+ библиотекам. Покрывает knowledge gap между датой отсечки LLM и текущими версиями фреймворков. Подробно описан в [разделе Context7](#context7--документация-по-требованию).

#### Единый контекст для LLM

```
┌─────────────────────────────────────────────────────────────────┐
│                      ПРОМПТ → DeepSeek V4 Pro                    │
├─────────────────────────────────────────────────────────────────┤
│                                                                   │
│  ┌─────────────────────┐  ┌──────────────────┐  ┌─────────────┐ │
│  │ Задача пользователя │  │ Код проекта      │  │ Документация│ │
│  │ (инструкция)         │  │ (Voyage+LanceDB) │  │ (Context7)   │ │
│  └─────────────────────┘  └──────────────────┘  └─────────────┘ │
│                                                                   │
│  Все три источника объединяются в один промпт. LLM видит:        │
│  • Что нужно сделать (задача)                                     │
│  • Как это уже сделано в проекте (код)                            │
│  • Как правильно использовать библиотеки (документация)            │
│                                                                   │
└─────────────────────────────────────────────────────────────────┘
```

### Нужен ли отдельный RAG-сервис

**Нет, не сейчас.** Для задач разработки хватает двух retrieval-слоёв:
- Собственный код покрыт Voyage + LanceDB
- Внешние библиотеки покрыты Context7

Отдельный RAG-сервис (с индексацией внутренней вики, контрактов, API-спецификаций, логов) имеет смысл только при появлении **новых доменов знаний**, не покрытых текущими инструментами. Добавление третьего retrieval-слоя поверх существующих не даст прироста качества для текущих задач, но увеличит сложность и стоимость индексации.

### Почему это названо «Code Indexing», а не «RAG»

Kilo Code позиционирует себя как целостный инструмент разработчика, а не как RAG-фреймворк. Пользователь не думает в терминах Retrieval/Embedding/Chunking — он просто задаёт вопрос на естественном языке, а система сама подтягивает релевантный код и документацию. `@kilocode/kilo-indexing` — это RAG-компонент, упакованный в плагин так, чтобы разработчику не нужно было знать, как он устроен внутри.

## Дополнительные компоненты

### LSP (Language Server Protocol)

**Включён:** `"lsp": true`

LSP обеспечивает навигацию по коду на уровне IDE: переход к определению (go-to-definition), поиск использований (find-references), диагностика ошибок. Работает через Tree-sitter (встроен в бинарный файл Kilo Code) и внешние language-серверы.

### Compaction (сжатие контекста)

**Настройки:**
- `"auto": true` — автоматическое сжатие при заполнении контекстного окна
- `"prune": true` — удаление старых результатов инструментов

Контекстное окно LLM ограничено (токенов). Когда диалог становится слишком длинным, Compaction сжимает историю: суммирует предыдущие действия, оставляя только существенную информацию. Это позволяет вести длинные сессии без потери связности.

### Агенты (Agents)

Kilo Code поддерживает мульти-агентную архитектуру — переключение между специализированными промптами:

| Агент | Назначение |
|---|---|
| **architect** | Проектирование архитектуры, планирование задач |
| **code-reviewer** | Проверка кода: качество, безопасность, производительность |
| **docs-specialist** | Написание и рецензия документации |
| **test-engineer** | Написание тестов, отладка, повышение покрытия |

### Speech-to-Text

**Модель:** `openai/whisper-large-v3-turbo`

Экспериментальная функция: голосовой ввод через OpenAI Whisper API. Позволяет диктовать задачи голосом вместо набора текста.

### Разрешения (Permissions)

Детальная система контроля доступа: каждое действие (чтение файлов, запуск команд, правка кода) может быть разрешено, запрещено или требовать подтверждения. Правила задаются через glob-паттерны.

## Примечания по развёртыванию

### Порядок запуска

```bash
# 1. Глобальная установка Kilo Code (один раз)
npm install -g @kilocode/cli

# 2. Настройка конфигурации (~/.config/kilo/kilo.json)
#    - Указать DeepSeek API ключ
#    - Настроить Voyage AI ключ для индексации
#    - Включить нужные MCP-серверы

# 3. Запуск
kilo
```

### Конфигурация в проекте

```jsonc
// .kilo/kilo.json (проектный уровень — переопределяет глобальный)
{
  "$schema": "https://app.kilo.ai/config.json",
  // Переопределение настроек для конкретного проекта
}
```

### Требования

- **Node.js** ≥ 20 (для MCP-серверов и npm-пакетов)
- **Дисковое пространство** ≈ 500 МБ для LanceDB-индекса (зависит от размера проекта)
- **Сеть** — доступ к API DeepSeek, Voyage AI, Context7
